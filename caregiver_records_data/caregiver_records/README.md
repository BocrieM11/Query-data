## 使用场景

本数据集为以下 AI 养老平台功能设计：

1. **LLM 语音转文字 + 结构化提取**
   - `record_text` → 口语化护工记录（日文/中文为主）
   - `vital_signs` / `meal_intake` / `mood` → 从文本中提取的结构化字段，可作为训练标签

2. **Dashboard 数据分析**
   - 同一 `resident_id` 多日连续记录 → 趋势图表（血压走势、体重变化）
   - 异常事件（category 04/05）→ 风险预警模型
   - 交接班（category 06）→ 工作流优化

3. **多语言支持**
   - 日语为主（日本养老市场），中文/英文用于多语言扩展验证

## 构建方法

### 总体流程

```
公开标准文档                    真实公开记录
(厚生労働省/DB36/DB50/CMS)    (学术论文/公开模板/新闻)
        │                              │
        ▼                              ▼
   templates.json              real_chinese_examples_extended.json
   (103 条模板)                 (13 条脱敏示例)
        │                              │
        ▼                              ▼
   build_final.py ────────▶  all_records.json  ◀── integrate_real.py
   (模板填充+多语言扩展)      (513 条最终数据集)
```

### 合成数据（500 条）

**第一步：模板构建**。从以下公开标准中提取护工记录模板，构建 `templates.json`（103 条）：

| 标准 | 适用地区 | 提取内容 |
|------|----------|----------|
| 厚生労働省介護記録標準フォーマット | 日本 | 日文口述式记录模板（食事·排泄·入浴·体位変換 等） |
| DB36/T 1582-2022 | 中国·江西 | 护理记录表（附录C）、日常健康记录表（附录F） |
| DB50/T 1421-2023 | 中国·重庆 | 照护记录单（附录C）、压疮预防记录（附录G）、交接班记录（附录L） |
| CMS MDS 3.0 | 美国 | 护理评估条目，转写为英文护理日志格式 |
| 养老机构跌倒应急预案 | 中国 | 跌倒事件处置模板 |

**第二步：模板填充**。运行 `build_final.py`，对每个模板进行：

- 随机分配 `resident_info`（老人池共 24 位虚构老人，每个 `resident_id` 对应的年龄、性别、护理等级、主要病症保持固定，模拟多日连续记录）
- 根据记录场景填充 `vital_signs`（数值基于各标准规定的正常/异常范围随机生成）
- 生成 `meal_intake`、`fluid_intake_ml`、`mood` 等辅助字段
- 确保每条 `record_text` 唯一（通过场景变体 + 随机细节实现去重）

**第三步：多语言扩展**。以日文模板为基础（日本养老市场规模最大），扩展中文和英文版本：

| 语言 | 条数 | 风格 |
|------|------|------|
| 日文 | 125（25%） | 口语化口述风格，模拟护工语音转文字场景 |
| 中文 | 250（50%） | 口语化口述风格，基于 DB36/T + DB50/T 本地化 |
| 英文 | 125（25%） | 护理日志书面风格，基于 CMS MDS 3.0 转写 |

### 真实数据（13 条）

独立于合成管线，从公开渠道收集真实/范例护理记录，经脱敏后通过 `integrate_real.py` 整合入主数据集：

| 来源渠道 | 条数 | 脱敏处理 |
|----------|------|----------|
| 学术论文（上海H社会福利院个案） | 2 | 去除机构名称、具体日期 |
| 公开护理日志/交接班/压疮模板范例 | 5 | 姓名已脱敏（模板自带XXX占位） |
| 养老机构公开事件记录 | 1 | 去除机构名称 |
| DB36/T + DB50/T 国家标准附录 | 5 | 标准文档本身不含个人信息 |

真实记录 `id` 范围为 `REC-REAL-0002` ~ `REC-REAL-0032`（23条），扩增记录 `id` 范围为 `REC-AUG-0000` ~ `REC-AUG-0189`（190条）。扩增记录基于真实记录的结构和风格生成，每条经过医学数值范围校验和去重处理。

### 质量保证

数据集经过自动化审计脚本 `audit_quality.py` 和 `fix_vital_signs.py` 检查修复：

| 检查项 | 结果 |
|--------|------|
| 完全重复文本 | 0 条 |
| `vital_signs` 字段合理性 | 全部在医学正常范围内 |
| 居民档案一致性 | 同一 `resident_id` 的属性无矛盾 |
| `record_text` 与结构化字段对应 | 数值一致 |
